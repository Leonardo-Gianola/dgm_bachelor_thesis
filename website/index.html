< 01 > Abstract

A Machine That Rewrites Itself

The Darwin Gödel Machine (DGM) is a self-improving coding agent that iteratively modifies its own source code and empirically validates each change on real-world software engineering benchmarks.


Unlike prior approaches that rely on hand-crafted improvements, DGM leverages large language models to propose targeted mutations to its own implementation — targeting reasoning prompts, tool usage, and patch application logic. Each modification is tested on a subset of real GitHub issues from SWE-bench Verified Mini, and improvements are retained only when they demonstrably increase resolution rate.


This thesis extends the original DGM framework with novel multi-fidelity schedulers (ASHA and Hyperband) that reduce evaluation cost while maintaining competitive performance, enabling faster exploration of the self-improvement search space.

⚙️
Self-Improvement

The agent modifies its own Python source code — prompts, tools, and reasoning chains — to improve future task performance.

🧬
Evolutionary Selection

Only code changes that improve benchmark accuracy are kept. Failed mutations are discarded, preserving a growing archive of elite agents.

📊
Empirical Validation

Every proposed change is tested on real GitHub issues (Django, Sphinx, etc.) — not simulated environments.

⚡
Multi-Fidelity Scheduling

ASHA and Hyperband rungs evaluate promising agents on progressively larger task subsets, cutting compute waste by up to 70%.

< 02 > Method

The DGM Loop

01
Initialization — Seed the Archive

The process begins with a hand-crafted baseline coding agent and a benchmark dataset. The initial agent attempts all tasks, establishing the starting accuracy score that all future generations must beat.

Initial Baseline: The seed agent achieves 20% accuracy on SWE-bench Verified Mini (12 out of 60 tasks resolved). It uses GPT-based reasoning with bash and file-editing tools. This single agent forms the root of the evolutionary archive.
02
Parent Selection — Choose a Candidate

From the archive of validated agents, a parent is chosen to serve as the base for the next mutation. Multiple selection strategies are available, balancing exploration vs. exploitation.

Selection Strategies:
• Random — uniform sampling from archive
• Score-Proportional — high-accuracy parents preferred via sigmoid weighting
• Tournament — k random candidates, best wins (k=3 default)
• Best — always pick the current top performer
• No-Darwin — always the latest commit (control baseline, no evolution)
03
Self-Modification — LLM Rewrites the Agent

An LLM (GPT-4 / Claude) analyzes the parent agent's failures and proposes targeted code changes. The mutation targets specific components: reasoning prompts, tool implementations, or the workflow logic.

Mutation Process:
1. Diagnosis — analyze which tasks failed and why (error logs, patch mismatches)
2. Proposal — LLM suggests specific code changes with rationale in JSON format
3. Application — changes are applied to a fork of the parent agent's source
4. Compilation check — invalid Python is immediately discarded before evaluation

Targets include: system prompts, AST-aware editing logic, test-verify loops, context management
04
Evaluation — Test on Real Benchmarks

The mutated agent is deployed against a subset of real GitHub issues. It must find and apply the correct patch to make previously-failing tests pass. Accuracy score drives the selection decision.

Evaluation Metrics:
• accuracy_score = resolved_tasks / total_submitted_tasks
• total_resolved_ids — list of successfully patched issues
• empty_patch_ids — agent gave up (generated no change)
• token_usage_total — compute cost tracking

Multi-fidelity schedulers (ASHA, Hyperband) evaluate on progressively larger subsets — stage_small → stage_medium → full — killing weak agents early.
05
Archive Update — Keep the Best

If the mutated child achieves a higher accuracy than its parent (or passes the multi-fidelity promotion threshold), it is added to the archive. Poor performers are discarded. The archive grows with elite agents only.

Archive Management:
• Agents stored as full Python source snapshots
• Each entry tagged with: accuracy_score, generation, parent_id, token_cost
• Archive enables branching evolution — future generations can descend from any elite
• Tie-breaking by token efficiency (fewer tokens = preferred when scores equal)

The loop then repeats from Step 2 — selecting a new parent from the updated archive.
Step 1 of 5

Evolution Schedulers

Baseline
Single agent, no evolution. Evaluates the initial agent across all tasks without self-improvement. Used as a control condition.
Genetic Algorithm
Standard evolutionary loop. Each generation produces one child via mutation. Configurable parent selection strategy (random, tournament, best, score-proportional).
ASHA
Asynchronous Successive Halving. Evaluates children on small task subsets first, promoting only top performers to larger, more expensive evaluations.
Hyperband
Extends ASHA with structured rung budgets (rung0 → rung1 → rung2). Maximizes compute efficiency across multiple fidelity levels simultaneously.
< 03 > Architecture

Two-Level Improvement Loop

Level 1 — Object Level

The coding agent solves real GitHub issues. Given a problem description and failing tests, it reasons, explores the codebase, writes patches, and validates fixes. This is the "work" the agent is evaluated on.

Level 2 — Meta Level

DGM uses a separate LLM to analyze the coding agent's failures and propose improvements to the agent's own Python source code — its prompts, tools, and reasoning workflow. This is the "self-improvement" layer.

Example Task

django__django-11815
Migrations auto-detector misses nested class model changes
▸
Problem: Django's migration auto-detector fails to detect model changes when models are defined as inner classes. The auto-detector traverses the model graph but does not recurse into nested class definitions, causing migrations to be silently skipped.

Why it's hard: Requires understanding Django's metaclass internals, the migration state machine, and the exact traversal order of the auto-detector's model registry. The agent must both locate the bug and write a semantically correct AST patch without breaking other migration paths.

Validation: FAIL_TO_PASS: tests.migrations.test_autodetector.AutodetectorTests.test_nested_model_changes

What the Agent Learned

Each generation the agent proposes targeted changes to its own source code. Here's what changed — and why it mattered.

Generation 0 · Seed
Initial Coding Agent — Baseline
A hand-crafted coding agent using GPT reasoning with bash and file-edit tools. Attempts SWE-bench tasks sequentially with a fixed reasoning prompt. No self-awareness of its own failures or patterns.
20.0% accuracy 12/60 resolved GPT-4o-mini
Generation 1 · First Mutation
Atomic Patch Application + AST-Aware Editing
The meta-LLM diagnosed that the baseline agent frequently produced malformed patches — diffs that looked correct but failed to apply cleanly due to whitespace and context line mismatches. The mutation introduced atomic transactional editing: the agent now previews its edit, verifies the target context exists, then applies — rolling back on any mismatch. AST-aware editing was added to handle Python-specific constructs without corrupting indentation.
50.0% accuracy +30 pp GPT-4o-mini 10-task subset
Future Generations · Ongoing
Further Self-Improvements in Progress
Potential targets identified by the diagnosis system: improved context management for long files, smarter test-verify loops, multi-step reasoning traces, and better handling of import resolution errors in patched modules.

Standing on Giants' Shoulders

Original Gödel Machine (Schmidhuber 2003)
The theoretical predecessor: a self-referential agent that rewrites itself only when it can formally prove the rewrite improves expected utility. DGM replaces the formal proof with empirical validation.

DGM — Zhang et al. 2025
The original Darwin Gödel Machine paper from Sakana AI. Demonstrates open-ended self-improvement on SWE-bench and Polyglot. This thesis extends their framework with multi-fidelity scheduling.

SWE-bench (Jimenez et al. 2023)
Benchmark of real GitHub issues from 12 popular Python repositories. Tasks require understanding codebases, reasoning about bugs, and generating valid patches that make failing tests pass.
benchmark
Hyperband (Li et al. 2018)
Multi-fidelity hyperparameter optimization. Allocates more resources to promising configurations while early-stopping poor ones. This thesis applies Hyperband to agent evaluation budgets.
scheduling
ASHA (Li et al. 2020)
Asynchronous Successive Halving — removes synchronization barriers from Hyperband, enabling continuous agent promotion without waiting for all siblings in a rung to complete.

LLM-Based Coding Agents
SWE-agent, Devin, OpenHands, and other systems demonstrate that LLMs with tool access can solve real software engineering tasks — providing the capability foundation DGM evolves.

< 08 > Conclusion

What We Found

Achievements

  • Demonstrated that a coding agent can double its accuracy in a single self-improvement generation — from 20% to 50% on SWE-bench mini.
  • Implemented and validated four scheduling strategies (Baseline, GA, ASHA, Hyperband) with a unified scheduling interface.
  • Showed that targeted mutations — specifically patch application logic — have disproportionately large performance effects.
  • Built a reproducible, extensible DGM framework with shared archive management, token-based tie-breaking, and benchmark caching.
  • Confirmed that empirical validation (running the agent on real tasks) is a practical proxy for theoretical proof-of-improvement.

Future Work

  • Run additional generations to observe whether improvement compounds or plateaus, and identify self-improvement phase transitions.
  • Extend multi-fidelity scheduling to support population-based training (PBT) with concurrent agent evolution branches.
  • Explore cross-benchmark transfer: agents trained on SWE-bench, tested on Polyglot without fine-tuning.
  • Investigate safety constraints: ensuring self-modified agents don't acquire unintended capabilities or bypass evaluation metrics.
  • Apply DGM to multi-agent systems where agents evolve both task skills and coordination protocols simultaneously.
